Requirements Specification: Catalog, Production Dashboard, Revenue & Growth Analytics, Designer Task Console
Companion document to Project_Atlas_Requirements.docx (Sections 1–10)

1. Executive Summary
The backend architecture defined in Project_Atlas_Requirements.docx produces and tracks an enormous amount of state — personalities, story universes, backlog items, agent pipeline status, revenue actuals, growth metrics, and now human merch-design tasks — but none of it is currently visible to a human. This document specifies a frontend application, the “Operator Console,” whose job is to make that backend state visible, searchable, and actionable for the small internal team running the platform.
The Console is scoped to five functional areas, matching how the underlying data model is already organized: a Content Catalog (personalities, story universes, backlog), a Production Dashboard (daily workflow and agent pipeline health), Revenue & Monetization (actuals against the daily floor, affiliate and sponsorship pipelines), Growth & Analytics (CTR, retention, and the daily learning loop), and a Designer Task Console (the human merch-production workflow defined in Section 10 of the main requirements document). It is built as an internal tool for a small team, not a public-facing product, which simplifies auth, permissions, and scale requirements considerably.
This is a requirements specification, not an implementation. It defines screens, data needs, and API/backend touchpoints in enough detail that a build can start directly from this document, cross-referenced against the data model and agents already defined in the main requirements document.
2. Goals, Users & Scope
2.1 Goals
Visibility. Give the team a single place to see what the system has discovered, what it's producing, what it's earning, and how it's growing — without querying Postgres/AGE directly.
Control. Let an operator intervene where the spec calls for human judgment: approving backlog items (Chief Editor Agent's role), overriding the evergreen vault pick on a forecasted revenue-floor miss (Section 8.11), pausing a sponsorship deal stage, or reviewing a designer's merch submission (Section 10 of the main requirements document).
Trust. Surface enough of the “why” behind a score or a diagnosis (e.g. why a video was flagged as a CTR problem vs. a hook problem, per Section 9.10, or why a merch submission was rejected on compliance grounds, per Section 10.5) that the team can sanity-check the agents rather than blindly trusting a number.
Speed. The daily revenue floor (Section 8.11), daily learning loop (Section 9.10), and time-sensitive merch briefs tied to event/festival windows (Section 10.1) all depend on same-day human or agent action — the Console must make today's anomalies and deadlines obvious within seconds of opening it, not buried in a report.
2.2 Users
Internal team tool. Initial user base is the small team operating Project Atlas — expected to be a handful of people (content lead, ops/producer, the person managing revenue/partnerships, and one or more human designers fulfilling merch tasks). No public or external accounts. A simple shared-login or lightweight per-person login is sufficient; this is explicitly not a multi-tenant, role-gated enterprise system at this stage, though designers are the first user segment that meaningfully needs a scoped view (their own tasks only) rather than full visibility — addressed in Section 8.
If the team grows, the data model (Section 7) already includes a generic user/session table that can later support per-role views without a redesign — noted as a forward-compatibility consideration, not a v1 requirement.
2.3 In Scope (v1)
Content Catalog: browse, search, and filter personalities, story universes, and the story_backlog.
Production Dashboard: daily workflow status across the six agent layers and the Temporal pipeline; today's publish queue.
Revenue & Monetization: revenue_actuals vs. the daily floor target; affiliate_links and sponsorship_deals pipelines.
Growth & Analytics: CTR, retention, subscriber conversion, and the daily learning/feedback loop diagnosis feed.
Designer Task Console: assigned merch task list, task detail with brief and Envato reference material, submission upload, and compliance review status (Section 10 of the main requirements document).
2.4 Out of Scope (v1)
Editing scripts, thumbnails, or any content asset directly in the Console — production stays inside the agent pipeline.
Direct SQL/raw-Cypher query tools for non-technical users remain out of scope — the Console stays a curated view, not a database client. A bounded, point-and-click Knowledge Graph Explorer is now in scope as of Section 12, which extends this principle to the graph rather than violating it.
Multi-tenant access and public/customer-facing views remain out of scope. Granular RBAC is now in scope as of Section 14 (Role-Based Access Control), which supersedes this v1 placeholder.
Mobile app — a responsive web app is sufficient for an internal team checking it from a laptop or occasionally a phone browser.
In-browser design/editing tools for designers — the Designer Task Console handles brief delivery and file submission, not artwork creation itself.

8. Designer Task Console
Surfaces the human merch-production workflow defined in Section 10 of the main requirements document (the Merch Task Brief Agent → Designer Assignment Agent → human Designer → Submission Review Agent → Upload Agent pipeline, with Envato Elements sourcing and the licensing compliance gate). This is the one section of the Console primarily used by a different user — the human designer fulfilling a task — rather than the content/ops/revenue team, so it is scoped as a focused, single-purpose view rather than a dense data console.
8.1 Designer Scoped View
Unlike the rest of the Console, this section should be usable by a designer who only needs to see their own work, not the full backend.
Default scope: own tasks only. A logged-in designer's “My Tasks” view filters designer_tasks to assignee_id = self by default. Ops/content team logins can switch to an all-designers view for workload balancing (Section 8.4).
No backend visibility required. Designers do not need, and should not be given by default, visibility into story_backlog scores, revenue_actuals, or other agents' internal data — only the fields on their assigned designer_tasks and merch_briefs records.

8.3 Task Detail View
Opened from the list. Combines the brief with any Envato-sourced reference material and the submission/review state.
Brief panel: product type, theme, archetype, target region, motif/key phrase, deadline, and the event/trend rationale behind the opportunity — pulled directly from merch_briefs so the designer sees the same reasoning the Merch Task Brief Agent used.
Envato reference panel: thumbnails and links for any envato_asset_references attached to the task (stock photos, mockup templates, fonts, graphic elements), each labeled with its usage_role (reference / mockup / incorporated_component) — making explicit to the designer which assets are starting material only versus assets the compliance gate (Section 10.5 of the main requirements document) expects to see meaningfully transformed in the final file.
Submission panel: file upload control for the finished artwork (and any requested source/working files), plus a required incorporated_description text field whenever the task has non-empty Envato references with usage_role = incorporated_component — the field is enforced as required at the UI level specifically because Section 10.5 makes the Submission Review Agent's compliance check dependent on this description existing and being substantive, not boilerplate.
Review status panel: read-only display of review_outcome and review_notes once the Submission Review Agent (or a human reviewer overriding it) has processed the submission — approved, sent back for revision with specific notes, or rejected with a reason. On approval, the live listing_url appears here once the Upload Agent publishes it.

If the Console's API is ever opened to third parties or other internal services outside this codebase, those integration points should use REST endpoints alongside tRPC rather than forcing tRPC on external consumers.
9.2 Data Access Pattern
Read path. Server components / tRPC queries read directly from PostgreSQL for relational tables (story_backlog, revenue_actuals, affiliate_links, sponsorship_deals, designer_tasks, merch_briefs) and from Apache AGE for graph traversals (personality relationships, story universe structure).
Write path (limited). The Console only writes back for the specific human-in-the-loop actions defined in this document: backlog approval (4.1), sponsorship stage advance (6.4), manual evergreen-vault pull (6.1), and designer task submission/review (8.3–8.4). It does not write production data, scores, or analytics — those remain owned by the agents per the Section 2.3 data-ownership principle in the main requirements document.
Caching. Dashboard and Growth views can tolerate a few minutes of staleness; use short-TTL server-side caching (e.g. 2–5 minutes) rather than real-time subscriptions for v1, given the daily (not real-time) cadence of the underlying feedback loops (Sections 8.11, 9.10). The Designer Task Console's deadline/notification logic (Section 8.5) is the one area that should not be cached beyond a minute or two, since it gates time-sensitive escalations.

12. Knowledge Graph Explorer
Adds the drill-down/explore capability the v1 spec explicitly excluded (Section 2.4) — but as a bounded, curated view, not the raw database client the v1 exclusion was actually guarding against. The distinction matters: this section extends the Console's existing principle of curated access to a new data shape (graph) rather than abandoning the principle.
12.1 Entry Points
From a detail view. Every Catalog detail view (Section 4.2) and Published Content row (Section 11.2) gets an “Explore in Knowledge Graph” action that opens the graph centered on that entity.
Standalone nav item. A top-level “Knowledge Graph” section with a search-to-center control, backed by the same Apache AGE-powered global search already used elsewhere in the Console (Section 3).
12.2 Visualization
Rendering. A force-directed graph component (e.g. react-force-graph or Cytoscape.js) rendering node types (Personality/Subject, Story, Archetype, Theme, Event, Region) and the Apache AGE relationship types from both the original model (INSPIRED, RIVALED, FAILED, REINVENTED, LED, MENTORED, INFLUENCED, ACQUIRED) and the Section 11 generalization (PREYS_ON, MIGRATES_TO, NESTS_IN, EXHIBITS, REQUIRES_SKILL, PROGRESSES_FROM).
Drill-down panel. Clicking a node opens a side panel reusing existing detail-view data (entity summary, linked published objects, score breakdown where the viewer's role permits it per Section 14) rather than introducing new backend logic.
Depth control. A 1-hop / 2-hop expand toggle, since the underlying graph is large (50,000+ personalities / 5M+ relationships per the main doc's Section 6.3 target scale) and an unbounded render would be unusable.
12.3 Data Access Pattern
The graph view is served by a small, fixed set of parameterized Cypher queries behind tRPC endpoints — getNeighborhood(entityId, depth) and searchEntities(query) are the only two needed for v1 — never an open query box. This is what keeps the “curated view, not a database client” principle intact while still answering the drill-down requirement.
12.4 Performance Guardrail
Cap server-side expansion at 2 hops or 150 nodes per request, whichever is hit first, with a “load more” action to page further rather than returning an unbounded neighborhood in one response.

Generalizes two compliance mechanisms that already exist but are buried inside agent logic — the Redbubble AI-content rules reference (main doc Section 10.7) and the Fact Verification Agent's advertiser-unfriendly/demonetization-trigger flagging (main doc Section 8.4) — into one cross-platform subsystem covering YouTube and Instagram as well, visible directly in the Console rather than only inferable from a pass/fail flag on a video.

13.2 Pre-Publish Compliance Gate
Before an item moves from Content Factory to the Publishing Queue (main doc Section 6.4 daily workflow), a compliance check runs against the active platform_policies rows: demonetization triggers, AI-disclosure requirements (YouTube's altered/synthetic content toggle, Instagram's AI label), and copyright-safe sourcing (already governed by main doc Section 2.5). The result becomes a new column on the existing Publish Queue view (Section 5.2) and the new Published Content Tracker (Section 11.2): Clear / Flagged / Needs Review.
13.3 Policy Change Monitoring
A scheduled job periodically re-checks each platform_policies.source_url for material change and opens a review task for an operator — it never silently auto-applies a detected change. This surfaces as a new “Policy Watch” view: every tracked policy, its last-verified date, and any pending change alert.
13.4 Audit Trail
Every compliance check result — pass, flag, or human override — is logged with a timestamp and the specific policy version it was checked against, so a strike or takedown can be traced back to exactly what was known and verified at publish time. Exposed as a read-only, filterable audit log (by video/post, by channel, by policy area).
14. Role-Based Access Control
Replaces the v1 placeholder — “simple login with a designer/ops role flag” (Section 9.3) and “granular RBAC: out of scope” (Section 2.4) — with a real role model, needed now that the Designer Task Console (Section 8) is being opened to outside collaborators rather than internal staff only, and the team itself is expected to grow beyond a single Super User.

14.2 Designer / Visual Artist: Allow-List Field Model
Section 8.1 already states a designer “should not be given by default” visibility into agent internals. This section converts that intent into an explicit allow-list enforced at the API layer — not a UI-level filter that a determined user could route around by inspecting network requests.
Allowed, and only this: brief title / theme, product type, archetype tag, target region, motif / key phrase, deadline, Envato reference assets (thumbnails and links), the artwork submission control, and review_outcome / review_notes once a submission has been reviewed.
Never returned to a designer-scoped session, enforced at the API layer: agent names, opportunity / final / psychology / emotion scores, story_backlog or revenue_actuals rows, the Knowledge Graph, or the words “agent,” “AI,” “DS-Star,” or “KiloClaw” anywhere in UI copy or API payloads reaching that session.
Vocabulary. The ticket itself should read as an ordinary freelance creative brief — “Creative Brief,” “Reference Assets,” “Submit Artwork” — never “task generated by Merch Opportunity Agent.”
14.3 Enforcement Mechanism
API layer. tRPC middleware resolves the caller's role before a response is constructed and applies the field allow-list there — a designer-scoped query physically cannot retrieve a disallowed field, rather than retrieving it and hiding it in the UI.
Database layer (defense in depth). Postgres Row-Level Security policies on story_backlog, revenue_actuals, and other internal tables, scoped by role and channel, so a future API surface that bypasses the tRPC layer still can't leak this data.
14.4 Onboarding a New Designer
A Super User creates the account, assigns the Designer / Visual Artist role, and optionally restricts it to specific channels or product types. No backend training or context is required — the experience is identical to receiving a brief from any ordinary client and submitting finished artwork back, which is the entire point.
